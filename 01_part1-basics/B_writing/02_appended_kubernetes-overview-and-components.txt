1. コンテナが増えた世界で何が問題になったのか
第1章で見てきたように、
コンテナによってアプリケーションは
「環境から切り離された実行単位」として扱えるようになりました。

これは大きな前進でしたが、同時に新しい問題を生みます。

・実行単位が増える
・実行場所が固定されない
・常にどこかが壊れている前提になる

単体では正しく動くアプリケーションも、数が増え、場所が分散すると、
全体としては、「人が把握しながら運用する」やり方が簡単に破綻するようになります。

ここで直面したのは、「どう作るか」ではなく、
増え続ける単位を誰が、どうやって、成立させ続けるのか
という問題でした。

人が一つ一つ状態を確認し、必要に応じて操作する。
このやり方は、単位が少ないうちは機能します。

しかし、コンテナの世界ではこの前提がすぐに崩れます。


2.Kubernetes は何を解決しようとしているのか
Kubernetes は、「コンテナを動かすためのツール」ではありません。

Kubernetes が向き合っているのは、分散した実行単位を
人の手によらず、全体として成立させ続けることです。

そのために Kubernetes が行っているのは、

・コンテナの起動手順を覚えること
・サーバーを管理すること

ではありません。

Kubernetes が扱うのは、「あるべき状態」 です。

たとえば、

・このアプリケーションは何個動いていてほしいのか
・どこで動いていてほしいのか
・壊れたとき、どういう状態に戻っていてほしいのか

といった 状態の定義 を受け取り、それが成立するように動き続けます。

逆に言えば、

・アプリケーションの中身
・ビジネスロジック
・実装の詳細

といったものは、Kubernetesの責任範囲ではありません。

ここを切り分けて考えることが、Kubernetesを理解する最初のポイントです。


3.人が操作しないという設計思想
Kubernetes を初めて触った人が
戸惑う点の一つに、「なぜ、コンテナを1つずつ直接操作しないのか」
という疑問があります。

従来の運用では、

・起動する
・停止する
・再起動する

といった 「人が直接手を動かす操作」 が中心でした。

しかし、分散した環境ではこのやり方は長く持ちません。

なぜなら、

・状態は常に変わる
・操作した直後に別の変化が起こる
・人が追いかけられない

からです。

そこで Kubernetes は、操作ではなく状態を宣言するという設計を採ります。

「こうしてほしい」ではなく、「こういう状態であってほしい」
を伝え続ける。

Kubernetesは、現在の状態と理想の状態を常に比較し、
その差分を自動的に埋め続けます。

ここで重要なのは、人が毎回状況を見て判断しながら
操作を切り替える世界ではなくなった、という点です。


4.Kubernetes の全体像 ― ControlPlaneとNodeの役割
― Control Plane と Node の役割 ―

この考え方を実現するために、Kubernetes は明確な役割分担を持っています。
大きく分けると、

・ControlPlane
・Node（Worker Node）

の二つです。

ControlPlane は、「こうあるべき状態」を理解し、
それを成立させるための判断を行います。

一方で Node は、実際にコンテナを動かす場所です。

ここで注意したいのは、ControlPlaneがNodeに対して
「命令」しているわけではないという点です。

Kubernetesの設計では、

・状態は API を通じて共有される
・各コンポーネントはその状態を見て、自律的に動く

という前提が置かれています。

この構造によって、

・一部が壊れても
・全体が止まらず
・状態が戻り続ける

という性質が生まれます。


5.主要コンポーネントは何を担当しているのか
Kubernetes には、複数の主要コンポーネントが存在します。

ここで大切なのは、名前を覚えることではありません。
見るべきなのは、「なぜこの分担になっているのか」です。

・APIServerは、状態の窓口として振る舞います
・Schedulerは、実行場所を決めます
・Controllerは、状態がずれたときに戻します
・kubeletは、Node上で状態を成立させます

それぞれが一つの責務だけを持ち、それ以上のことをしません。

この分離によって、

・一部が壊れても
・クラスタ全体が停止せず
・影響が限定される

という構造が保たれます。

ここでも、「便利だから増えた機能」ではなく、
壊れ続ける世界に耐えるための分割であることが重要です。


6.Kubernetes は OS ではない
Kubernetes はしばしば「分散OS」と表現されることがあります。

この表現は、複数のマシンを一つのまとまりとして扱えるという意味では、
直感的には分かりやすいかもしれません。

しかし、この理解のまま進むと、いくつかの重要な点を見誤ります。

まず、OSが担っている役割を整理すると、

ハードウェア資源の管理
プロセスの実行と制御
メモリやネットワークの直接的な扱い

といった、「実際に動かすための基盤」を提供することにあります。
一方で Kubernetes は、これらを直接扱っているわけではありません。

Kubernetes が行っているのは、どのような状態であってほしいかを受け取る
現在の状態と比較し、差分を埋めるように動き続けるという、「状態の維持」に関する制御です。

つまり Kubernetes は、

プロセスを起動する存在ではなく
サーバーを管理する存在でもなく
ハードウェア資源を直接扱う存在でもありません

これらはすべて、OSやその下の仕組みが担っています。
ここで重要なのは、Kubernetes は「上に乗る仕組み」であるという点です。

OSの上で動き、コンテナランタイムを通じて実行を委ね
ネットワークやストレージの詳細は抽象化された形で扱う

この構造を理解していないと、Kubernetes がすべてを解決してくれる
OSやネットワークを知らなくてもよいという誤解につながります。

実際には、Kubernetes は「責任を肩代わりしている部分」と
「前提として残っている部分」を明確に分けています。

この違いは、次章以降で扱う実行環境や、その下にあるOS・ネットワークの話と接続することで、
よりはっきり見えてきます。


7. この章のまとめ ― 次に見える問題
ここまでで、コンテナによって切り出された実行単位を、
どのように成立させ続けるかという問題に対する一つの答えが見えてきました。

Kubernetes によって、単位は自動的に配置され、
状態は常に監視され、ズレは仕組みによって修正されるという構造が成立します。

つまり、「制御」は人の手から仕組みへと移されたと言えます。
しかしここで、次の問いが自然に浮かびます。

それは、「その単位は、どこで動かすのか」という問題です。

コンテナは場所から切り離され、
Kubernetes はその配置を制御できるようになりました。

だからこそ、どの環境で動かすのか
どこまでを自分たちで制御するのか
どこを仕組みに任せるのか

という判断が、避けて通れなくなります。

この選択は、

コスト
可用性
運用の責任範囲

といった、システム全体の前提に関わります。

つまりここで初めて、「どこで動かすか」という問題が、
単なる環境選択ではなく、設計の一部として現れます。

次の章では、クラウドとオンプレミスという二つの環境を通して、
この「前提条件の違い」を整理していきます。

▲内部ルール
・マニフェストは出さない
・YAML は書かない
・kubectl コマンドは出さない
・数値・性能比較はしない
・常に「なぜこの設計なのか」に戻る
